Common Interview Questions & Concepts:

Time Complexity yeh batati hai ki ek algorithm input size badhne par kitna zyada time lega run hone mein. Yeh actual seconds mein time nahi batati, balki time ka 'growth rate' batati hai input ke respect mein. Hum isko 'Big O Notation' ka use karke express karte hain.

Example: Agar aapko ek list mein koi number search karna hai aur aap ek-ek karke har number ko check karte hain (Linear Search), toh worst case mein aapko poori list (n elements) check karni pad sakti hai. Iski Time Complexity O(n) hogi (Order of n). Agar list ka size double hoga, toh worst-case time bhi lagbhag double ho jayega.

Kyun important hai? Kyunki yeh humein alag-alag algorithms ko compare karne mein madad karti hai ki kaunsa bade inputs ke liye zyada efficient (tez) hoga.

Big O Notation ek mathematical tareeka hai Time ya Space Complexity ke 'Upper Bound' (worst-case scenario) ko describe karne ka. Yeh batata hai ki algorithm ka performance input size (n) badhne par kaise grow karega, constant factors aur lower-order terms ko ignore karte hue.

Common Big O Notations (Fastest se Slowest):

  • O(1) - Constant Time: Run time input size pe depend nahi karta. (e.g., Array mein index se element access karna, Stack mein push/pop). Bahut accha!
  • O(log n) - Logarithmic Time: Run time input size ke logarithm ke hisab se badhta hai. Har step mein problem ka size kam hota jaata hai. (e.g., Binary Search). Bahut accha!
  • O(n) - Linear Time: Run time input size ke directly proportional hota hai. (e.g., Linear Search, Array/List ko traverse karna). Theek hai.
  • O(n log n) - Linearithmic Time: Efficient sorting algorithms ismein aate hain. (e.g., Merge Sort, Quick Sort average case). Accha hai.
  • O(n2) - Quadratic Time: Run time input size ke square ke hisab se badhta hai. Nested loops mein aksar dikhta hai. (e.g., Bubble Sort, Selection Sort, simple nested loops). Chhote inputs ke liye theek, bade ke liye slow.
  • O(2n) - Exponential Time: Run time har naye element ke saath double ho jaata hai. Bahut slow, sirf bahut chhote inputs ke liye kaam karta hai. (e.g., Recursive Fibonacci ka naive solution).
  • O(n!) - Factorial Time: Aur bhi zyada slow. (e.g., Travelling Salesman Problem ka brute-force solution).

Space Complexity yeh batati hai ki ek algorithm input size badhne par kitni extra memory (RAM) use karega. Ismein input ko store karne ki space ke alawa, algorithm jo temporary variables, data structures, ya function call stack use karta hai, woh sab count hota hai.

Isko bhi Big O Notation mein express karte hain.

Example:

  • Agar aap ek function banate hain jo sirf kuch simple variables (int, float) use karta hai, chahe input kitna bhi bada ho, toh Space Complexity O(1) (Constant Space) hogi.
  • Agar aap input array ki size ka ek naya array banate hain, toh Space Complexity O(n) (Linear Space) hogi.
  • Recursive functions jo 'n' depth tak jaate hain, unki Space Complexity call stack ki wajah se O(n) ho sakti hai.

Time ke saath-saath Space bhi important hai, khaaskar jab memory limited ho.

Ek hi algorithm alag-alag inputs ke liye alag-alag time le sakta hai. Isliye hum complexity ko teen scenarios mein dekhte hain:

  • Best Case (Big Omega - Ω): Algorithm ka minimum possible run time. Yeh tab hota hai jab input sabse favorable ho. (e.g., Linear search mein element pehli position pe hi mil jaye - O(1)).
  • Average Case (Big Theta - Θ): Algorithm ka expected run time 'typical' ya random input ke liye. Isko calculate karna thoda mushkil ho sakta hai. (e.g., Quick Sort ka average case O(n log n)).
  • Worst Case (Big O - O): Algorithm ka maximum possible run time. Yeh tab hota hai jab input sabse unfavorable ho. Hum aksar Worst Case pe focus karte hain kyunki yeh performance ki guarantee deta hai. (e.g., Linear search mein element aakhir mein mile ya na mile - O(n); Quick Sort ka worst case O(n2)).

Interview mein aksar Big O (Worst Case) poocha jaata hai, lekin baaki cases ki understanding bhi zaroori hai.
